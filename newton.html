SystemStabilizer 상세
매 턴 system 메시지를 안정화해 Anthropic prefix-based 캐시를 보호합니다.
🔴 문제: RisuAI의 동적 system 주입
RisuAI는 매 턴 활성화된 Lorebook 항목을 system 메시지에 주입합니다.

결과: system이 매 턴 달라짐 → Anthropic 캐시는 prefix-based이므로 BP1부터 전체 무효화 → cache_create만 발생, cache_read=0
🟢 해결: canonical 저장 + delta 추출
첫 턴에 system을 canonical로 저장. 이후 턴에는 변경분만 추출해 마지막 user 메시지로 이동.

결과: system = canonical (고정) → BP1 캐시 유지 → cache_read 지속 누적

Turn 1: 최초 시스템 저장

paragraph-level diff 알고리즘

# system_stabilizer.py — _extract_delta()
canonical_paras = split_paragraphs(canonical)   # 기존 canonical
current_paras  = split_paragraphs(current_system) # RisuAI가 보낸 현재

added   = {p for p in current if current_count[p] > canonical_count[p]}
removed = {p for p in canonical if canonical_count[p] > current_count[p]}

# inject_at / inject_lore / inject_replace 감지
# → 수정된 단락은 delta에서 제외 (중복 주입 방지)
for r_para in removed:
    for a_para in added:
        if r_para in a_para or a_para in r_para:   # substring → inject_at
            matched_added.add(a_para)
        elif word_overlap(r_para, a_para) > 0.5:  # overlap → inject_replace
            matched_added.add(a_para)

pure_added = added - matched_added   # 진짜 새 Lorebook 항목만
delta_text = "\n\n".join(pure_added)  # → 마지막 user 메시지로

수명: 일회성 (해당 턴에만 user 메시지에 주입)
저장: DB에 저장되지 않음
예: 유저가 Lorebook에 새 NPC를 추가한 턴
Curator 로어 SAGA 자동 생성
출처: Curator가 에피소드 분석으로 자동 생성
목적: 메모리 강화 — NPC 정보를 구조화해 매 턴 제공

저장: DB 저장, 이후 매 턴 active_lore로 주입

BP2는 대화 중간점 assistant에, BP3는 마지막 assistant에 배치합니다.

Anthropic 최소 요구사항
  • cache_control이 붙은 블록은 최소 1024 tokens 필요
  • BP1 (system): TTL 1h (extended-cache-ttl 베타 활성)
  • BP2, BP3: TTL 5m (기본)
  • 캐싱은 prefix-based — 앞에서부터 연속 일치해야 히트

왜 동적 컨텍스트를 user 메시지에 넣는가

핵심 원리: Anthropic 캐싱은 prefix(앞에서부터의 연속 일치)로 작동합니다. system 메시지를 포함한 앞부분이 동일하면 캐시 히트. 변하는 내용(lorebook_delta, dynamic_suffix, 유저 입력)을 마지막 user 메시지에 집어넣으면, 캐시 경계 이전 prefix(BP1~BP3)는 항상 동일하게 유지됩니다.
system role에 동적 컨텍스트를 넣으면 안 되는 이유: system 메시지가 매 턴 바뀌면 BP1 마커가 붙어 있어도 prefix가 달라져서 캐시 미스. SystemStabilizer의 존재 이유입니다.

# BP1: system에 항상 붙임

if len(assistant_msgs) >= 2:

# post_turn.py — _calculate_importance()
score = 10  # base

if scene_type == "combat":      score += 40
elif scene_type == "event":     score += 35
elif scene_type == "exploration": score += 15

if key_event:                   score += 30
if npcs_mentioned:              score += 10 * min(2, len(npcs))  # 최대 +20

return min(score, 100)  # 캡 100, 중요도≥50 시 high-importance 로깅

# 1단계: 엑스트라 필터링 (unnamed extras 제거)
_EXTRA_PATTERN = r"^(마을|숲|동굴)?\s*(사람|여인|병사|상인|농부…)\s*\d*$"

# 2단계: 정규화 (조사/어미/경칭 제거)
def normalize(name):
    name = strip_parenthetical(name)   # "라쿤(Raccoon)" → "라쿤"
    name = strip_en_article(name)      # "the King" → "King"
    name = strip_jp_honorific(name)    # "루비아-san" → "루비아"
    name = strip_ko_particle(name)     # "루비아가" → "루비아"
    return name.lower()

# 3단계: 퍼지 매칭 (rapidfuzz token_set_ratio, threshold=88)

SAGA의 Curator는 구조화된 큐레이션을 합니다:
• 서사를 구간별로 분리해서 초기 이벤트도 보존
• NPC를 개별 엔티티로 추적해서 감정 변화/관계 변화 기록
• 복선을 우선순위 태깅해서 회수 시점까지 유지

30턴이 지나도 "Turn 5에서 루비아가 뺨을 붉혔다"를 정확히 기억합니다. 슬라이딩 윈도우로 메시지가 잘려도 이 큐레이션은 DB에 남아서 다시 주입됩니다.

예상 질문 대비

Q: ChromaDB와 SQLite를 왜 같이 씁니까?
A: ChromaDB는 의미적 유사도 검색(벡터), SQLite는 turn_log / lore / NPC 관계 같은 구조화 쿼리용입니다. 에피소드 검색은 벡터가 필요하고, "이 세션의 모든 NPC 목록"은 구조화 DB가 빠릅니다. 두 가지 요구사항이 동시에 있어서 이중 저장을 택했습니다.
Q: Letta를 쓰는 이유가 뭡니까?
A: Curator는 10턴마다 실행되는데, 에이전트가 이전 큐레이션 결과를 기억해야 서사 연속성이 생깁니다. Letta Memory Block은 에이전트가 자체 메모리를 보유합니다. Letta 불가용 시 DirectLLMCuratorAdapter로 자동 전환하는 fallback도 있습니다.
Q: RRF를 쓴 이유가 뭡니까?
A: 단순 벡터 유사도만 쓰면 최근 에피소드가 밀릴 수 있고, 최신 순으로만 하면 오래된 중요 사건을 놓칩니다. RRF로 최근성(1.2) / 중요도(1.0) / 의미적 유사도(0.8) 세 신호를 합산하면 균형 잡힌 에피소드 선택이 됩니다.
Q: 캐시 히트율 85~90%는 어떻게 측정했습니까?
A: Anthropic API 응답의 usage.cache_read_input_tokens와 usage.cache_creation_input_tokens를 매 턴 로깅했습니다. cache_read / (cache_read + cache_create)로 히트율을 계산했습니다. Turn 2~21 구간의 실측값입니다.
