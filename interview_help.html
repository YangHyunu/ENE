문제: RisuAI는 매 턴 system 메시지에 Lorebook을 동적 삽입합니다. Anthropic 캐시는 prefix-based이므로 system이 1자라도 바뀌면 모든 BP가 무효화됩니다.
SAGA 해결책: SystemStabilizer로 system을 고정하고, 변경분(delta)은 마지막 user 메시지에 prepend합니다. 모든 BP가 캐시 영역 뒤에 위치합니다.

Prefix-based 캐싱

Anthropic 캐시는 프롬프트 앞에서부터 연속 일치하는 prefix를 캐시합니다.
cache_control 마커가 있는 메시지까지의 prefix가 이전 요청과 동일하면 히트, 중간에 하나라도 바뀌면 그 지점부터 미스입니다.

cache_read

$0.10/MTok

이미 캐시됨. 정가 대비 90% 할인

cache_creation

$1.25/MTok

새로 캐시 생성. 정가 대비 25% 프리미엄

input_tokens

$1.00/MTok

BP 이후 토큰. 캐시 대상 아님

최소 토큰 임계값: Claude Haiku 4.5는 최소 4,096 토큰이어야 캐시가 생성됩니다. 미달 시 cache_control을 달아도 조용히 무시됩니다. cache_creation: 0이면 프롬프트 길이를 의심하세요.

동적 컨텍스트를 user 메시지에 넣는 이유

❌ 잘못된 배치

system: [시스템프롬프트 BP1]
         [동적컨텍스트]  ← 매 턴 변경됨!
messages: [...BP2...BP3...]

_call_anthropic()이 모든 role: "system" 메시지를 body["system"] 배열로 호이스팅합니다. 동적 컨텍스트가 system 앞으로 올라가 BP1 prefix를 변경 → 전체 캐시 무효화

✅ 올바른 배치

system: [시스템프롬프트 BP1]  ← 불변
messages: [...BP2...BP3...,
  "동적컨텍스트\n\n유저입력"] ← BP 뒤

마지막 user 메시지에 prepend. 모든 BP 뒤에 위치하므로 BP1~BP3 캐시에 영향 없음. 동적 컨텍스트는 매 턴 바뀌어도 캐시 히트에 영향을 주지 않습니다.

user 마지막 메시지 구조

[User:                                              ← 캐시 외 (매 턴 변경)
  --- SAGA Context Cache ---
  md_prefix           # 안정적 세계관 정보
  --- Active Lorebook ---
  lorebook_delta      # SystemStabilizer가 추출한 새 Lorebook
  --- SAGA Dynamic ---
  dynamic_suffix      # 에피소드 요약 + 라이브 상태

  "유저의 실제 입력"
]

Delta 추출 알고리즘

paragraph-level diff

system 메시지를 \n\n 기준으로 단락 분리 후 Counter로 비교합니다. 현재에만 있는 단락 = Lorebook delta.

inject_lore 감지

수정된 단락(기존 단락의 substring 또는 단어 50%+ 겹침)은 delta에서 제외합니다. inject_at/inject_replace 오탐 방지.

Jaccard 유사도 기반 canonical 교체

Jaccard = |교집합| / |합집합|   (단락 집합 기준)

similarity ≥ 0.30 → 같은 캐릭터, canonical 유지 (delta만 추출)
similarity < 0.30 → 다른 캐릭터 감지 → canonical 교체 (BP1 캐시 재생성)

SQLite KV 저장

canonical_system_prompt

첫 턴의 system 메시지 전체. 이후 매 턴 이 값을 Claude에 전달합니다. BP1의 안정성 보장.

canonical_system_hash

MD5 해시로 빠른 비교. 해시가 같으면 delta 추출 없이 즉시 반환합니다.

Cache Warming

TTL 만료를 방지하여 세션 재개 시에도 캐시 히트를 보장

세션 타임라인

준비됨

기본 파라미터

Warming 간격4.5분
최대 횟수4회
요청 크기max_tokens=1
Idle 중단 임계값10분
캐시 TTL5분 (마지막 사용 기준)

Warming 요청이 경량인 이유

max_tokens=1로 설정하면 캐시 히트 확인만 하고 실제 응답은 생성하지 않습니다. TTL이 사용 시점 기준 5분으로 리셋되므로, 4.5분 간격으로 ping을 보내면 무기한 유지할 수 있습니다.

Idle 10분 초과 시 자동 중단: 사용자가 세션을 떠났다고 판단, warming loop를 종료합니다.

비용 분석

Warming 비용 vs 절감 효과: Warming 1회 = cache_creation 비용 (수천 토큰 × $1.25/MTok). 하지만 세션 재개 시 전체 캐시를 재생성하는 것보다 훨씬 저렴합니다. 세션 재개가 잦은 RP 환경에서는 Warming이 항상 이득입니다.

발표 스크립트

AI 엔지니어 면접관 대상 — 기술 수준에 맞는 구두 설명 대본

Step 1 — 문제 정의 (30초)
"제가 만든 SAGA는 RisuAI의 리버스 프록시입니다. RisuAI는 매 턴 system 메시지에 Lorebook, 즉 세계관 데이터를 동적으로 삽입하는데요, Anthropic의 prompt caching이 prefix-based라서 system이 1자라도 바뀌면 모든 캐시 breakpoint가 무효화됩니다. 결과적으로 14,000 토큰짜리 system prompt를 매 턴 full price로 지불하게 됩니다."
핵심 용어: prefix-based caching, breakpoint invalidation, system prompt hoisting
Step 2 — 핵심 해결책: 3-BP 전략 (60초)
"제가 설계한 해결책은 3개의 명시적 캐시 브레이크포인트입니다. BP1은 system prompt에, BP2는 대화 중간 assistant에, BP3는 직전 턴 마지막 assistant에 배치합니다.

Anthropic 캐싱은 prefix-based입니다. 즉, 프롬프트 앞에서부터 cache_control 마커까지 연속으로 동일하면 캐시 히트가 됩니다. 자동 캐싱에 의존하면 system 메시지가 매 턴 바뀔 때 prefix 자체가 변경되어 전체 캐시가 무효화됩니다. 수동 3-BP 방식은 SystemStabilizer로 system을 고정하고 BP를 명시적으로 배치해서, 50턴에서도 97.3% 히트율을 유지했습니다."
강조 포인트: prefix 불변성이 핵심. SystemStabilizer가 system을 고정 → 전체 prefix 안정화
Step 3 — SystemStabilizer 설명 (60초)
"BP1이 안정적으로 히트하려면 system 메시지가 절대 바뀌면 안 됩니다. 그래서 만든 게 SystemStabilizer입니다.

첫 턴의 system을 'canonical'로 SQLite에 저장합니다. 이후 매 턴, 현재 system과 canonical을 paragraph-level diff로 비교해서 새로 추가된 Lorebook 단락만 'delta'로 추출합니다. canonical은 그대로 Claude에 전달하고, delta는 마지막 user 메시지에 prepend합니다.

캐릭터가 바뀌면 어떻게 되냐고요? Jaccard 유사도가 0.30 미만이면 아예 다른 캐릭터로 판단해서 canonical을 교체합니다."
구현: saga/system_stabilizer.py — _extract_delta(), _should_update_canonical()
Step 4 — Dynamic Context Injection 설명 (30초)
"동적 컨텍스트를 왜 user 메시지에 넣냐는 질문을 받을 수 있습니다. 처음에는 저도 system에 넣었는데요, FastAPI 서버가 모든 role: system 메시지를 body[\"system\"] 배열로 호이스팅합니다. 동적 컨텍스트가 system 앞으로 올라가버려서 BP1 prefix를 변경하고 전체 캐시가 깨졌습니다.

user 마지막 메시지에 prepend하면 모든 BP 뒤에 위치하므로 BP1~BP3가 전혀 영향받지 않습니다."
실수 경험: 시행착오 #1 — system role 삽입으로 cache_create 매 턴 증가, user prepend로 해결
Step 5 — Cache Warming 설명 (20초)
"마지막으로 Cache Warming입니다. Anthropic 캐시 TTL이 5분이라서, 사용자가 잠시 자리를 비우면 캐시가 만료됩니다. 4.5분마다 max_tokens=1의 경량 요청을 보내서 TTL을 리셋합니다. 최대 4회, idle 10분 초과 시 자동 중단합니다."
포인트: max_tokens=1로 응답 생성 비용 최소화, TTL 리셋만 목적
Step 6 — 예상 질문: "왜 RisuAI가 직접 안 하나요?" (30초)
"RisuAI는 오픈소스 RP 클라이언트인데, Anthropic API를 직접 지원하지만 prompt caching 최적화는 구현되어 있지 않습니다. automaticCachePoint 기능이 있지만 마지막 user 메시지 3개를 기준으로 하는 방식이라 장기 대화에서 system prompt 캐싱이 안 됩니다. 리버스 프록시 레이어에서 투명하게 최적화하는 것이 사용자 경험을 해치지 않는 가장 깔끔한 방법이었습니다."
Step 7 — 예상 질문: "캐시 히트율을 어떻게 검증했나요?" (20초)
"Anthropic API 응답의 usage 필드에서 cache_read_input_tokens, cache_creation_input_tokens, input_tokens를 매 턴 로깅합니다. 히트율 = cache_read / (cache_read + cache_creation + input_tokens). 50턴 벤치마크를 직접 실행해서 자동 캐싱 방식과 비교했습니다."
데이터: tests/bench_report_3bp.json, tests/bench_prompt_caching.py
